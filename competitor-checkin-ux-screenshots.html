1

Tap-a-row, green-checkmark confirmed state

The staff roster is a vertical list of attendee rows; each row carries a per-row check-in control (button, circle toggle, or segmented pill) and a green checkmark / filled-green state is the universal 'Checked-In' confirmation. The action is bundled into the row rather than living on a separate screen.

Used by: Mindbody ABC Glofox Mariana Tek WellnessLiving Vagaro PushPress Momence Walla GymMaster TeamUp

→ For Wix: Make the Owner App booking list the primary check-in surface and put a single thumb-reachable tap target on every row, with a green check as the only 'done' signal. Do not build a separate check-in screen the owner has to navigate into. A segmented No-Show | Check-In pill (Vagaro) or a circle toggle (WellnessLiving/Glofox) both work; for one-handed phone use a single tap-to-check-in with swipe-to-no-show is the most economical.

2

Color-as-status reinforced by an all-caps/text label

A small, consistent semantic palette communicates state: green=granted/checked-in/success, red=denied/blocked/failed, amber-yellow=owing-but-allowed warning, blue/indigo=in-progress or alert, grey=offline/ineligible. Critically the better implementations never rely on color alone — they pair it with an explicit text label (Kisi 'UNLOCKED', GymMaster 'Denied: Overdue Account', Glofox 'Access Granted').

Used by: Kisi GymMaster WellnessLiving ABC Glofox Mariana Tek Vagaro Momence

→ For Wix: Adopt a 4-state status vocabulary (granted/checked-in = green, blocked = red, warning/owing = amber, ineligible/expired = grey) and always render an accompanying label for accessibility and at-a-glance reading on a small screen. Reuse Wix Design System status colors but lock the meaning so the same green means the same thing on the roster, the kiosk, and any door event.

3

Eligibility surfaced inline BEFORE the check-in tap

Rather than failing after the tap, rosters expose membership/pass/payment/waiver status on the row itself: remaining visit counts, pack balance with low-balance alerts (Mindbody), 'Waiver Not Signed' tags (Mariana Tek/Walla), greyed-out ineligible passes you can still see but not select (TeamUp), and pre-completion blocker prompts. Denials are explained with a specific human-readable reason, never a generic 'access denied'.

Used by: Mindbody Mariana Tek TeamUp WellnessLiving GymMaster Walla ABC Glofox Momence

→ For Wix: Each booking row should carry an eligibility chip (valid plan / pass count / expired / payment due / unsigned form) so the owner sees the problem before tapping. When a check-in is blocked, show the exact reason and the resolve action (collect payment, sign form, override) in the same place — mirror TeamUp's greyed-but-visible ineligible option and GymMaster's explicit denial reasons rather than hiding state.

4

Inline override / resolve-in-place for the failure path

The blocked case is treated as a first-class flow, not a dead end. Staff can capture an electronic waiver signature on the row (Walla, Mariana Tek), 'Check-in & Pay' from the same panel (WellnessLiving), register-for-free / bypass membership (TeamUp), or override plan rules in staff (non-kiosk) mode (PushPress). Kiosk mode deliberately REMOVES override power for self-serve safety.

Used by: Walla WellnessLiving TeamUp Mariana Tek PushPress GymMaster

→ For Wix: Give the owner an inline resolve path from the blocked row: collect payment, sign a form, or override-and-check-in with one tap, without leaving the roster. Crucially, distinguish Owner mode (override allowed) from a future kiosk/self-serve mode (override hidden, must see staff) — PushPress's PIN-locked kiosk that disables overrides is the model for Phase 2.

5

Two identification rails: type-ahead name search OR scan a code

Self-serve and walk-in flows resolve identity two ways: (a) type-ahead name/phone/email search where results appear after a few characters (Mindbody, GymMaster, PushPress, Glofox), and (b) scan a credential — QR code, Code-39 barcode, key fob, or Apple Wallet pass (Vagaro QR+Wallet, Glofox auto-barcode from phone number, GymMaster dynamic QR, Walla QR). GymMaster uses one universal field that accepts name, phone OR email with no mode switch.

Used by: Mindbody GymMaster PushPress ABC Glofox Vagaro Walla WellnessLiving TeamUp Momence

→ For Wix: For the Owner App add-a-walk-in flow, lead with one universal type-ahead search field (name/phone/email, no mode toggle, results after ~2 chars) and a '+' to add a new client mid-session. For Phase-2 kiosk and the member's phone, plan a QR/Wallet credential (and the Kisi tap) — Wix can auto-generate a per-member QR like Glofox does from existing contact data rather than issuing physical cards.

6

Door event = automatic check-in via Kisi/Brivo with real-time status sync

Almost everyone outsources physical door hardware to Kisi or Brivo and syncs membership status in real time so access is granted/revoked instantly on plan change. The door-open event auto-records the arrival and auto-marks attendance, collapsing 'unlock' and 'check-in' into one event. Access is scoped by access groups, opening-hours schedules, or a booking time window (e.g. open 10 min before, 59 min after).

Used by: Mindbody ABC Glofox Mariana Tek WellnessLiving TeamUp Kisi GymMaster PushPress Momence

→ For Wix: Treat the Kisi door integration as a check-in source, not a separate system: a door unlock should post an arrival to the same booking and flip the same green check the owner would tap. Model access as Wix Bookings status + a time-window rule (before/after session start, TeamUp's plain-English summary) and surface a single 'allow blocked/owing members' toggle for the failed-payment edge case. The owner should never reconcile door events manually.

7

Roster header KPI band + status-segmented tabs/filters

Above the rows sits a summary band of counters (Attendance, Check-Ins, Waitlist — PushPress; capacity 30/30 — Mariana Tek) and the roster is segmented by status either as tabs (All / Checked-in / Reserved / Waitlist / No-Show / Late Cancel — PushPress; Check In / Waitlist / Add-ons — Mariana Tek) or collapsible sections. Bulk actions ('Check-in Everyone', 'Check-In All', select-all + Actions) clear a whole class in one tap.

Used by: PushPress Mariana Tek ABC Glofox Vagaro Mindbody

→ For Wix: Top the Owner App roster with a compact fill/attendance counter and offer a status filter (at minimum All / Checked-in / Not arrived / No-show) plus a bulk 'Check in all' for the common case where the whole class shows up. Keep it thumb-reachable; on a phone a horizontal scrollable filter chip row beats desktop-style tabs.

8

Hardware-free / 'any tablet becomes a kiosk' self check-in

Self check-in rarely requires proprietary hardware. Kiosks are either a web link opened in a browser after one admin login (Glofox, TeamUp, Momence) or the staff app flipped into a PIN/credential-locked Kiosk Mode (Mariana Tek, PushPress, Mindbody Check-In app, GymMaster). The kiosk reuses the staff roster visual sized up, plus a confirmation modal showing spot/add-ons/resource before 'Done'.

Used by: ABC Glofox TeamUp Momence Mariana Tek PushPress Mindbody GymMaster Vagaro

→ For Wix: For Phase-2 kiosk, reuse the same Owner App roster code in a locked-down Kiosk Mode (PIN to exit, overrides disabled, restricted nav) rather than building a separate app — this matches Mariana Tek and PushPress and keeps one visual language. The kiosk confirmation should echo the customer-relevant details (class, time, spot/resource if any, remaining passes) before a single 'Done', like Mindbody showing the room and Mariana Tek showing the spot.

9

Confirmation feedback: modal/full-screen wash + audio cue

Success and failure get loud, glanceable feedback designed for a noisy front desk and a queue: full-screen green wash (GymMaster), color-coded confirmation popups with green/red headers where the red header is tappable to reveal the reason (WellnessLiving), and success/fail audio chimes (WellnessLiving, GymMaster, Vagaro configurable). The confirmation surfaces photo/name/service/time.

Used by: GymMaster WellnessLiving Vagaro Mariana Tek Mindbody

→ For Wix: On the owner's phone a lightweight inline toast + the row flipping green is enough; reserve the big full-screen / audio confirmation for Phase-2 kiosk and member self check-in where the actor is at arm's length and unattended. Make a failed check-in's reason one tap away (WellnessLiving's tappable red header) rather than a dead toast.

10

Roster row as a relationship/recognition surface (photo + context tags)

Higher-end boutique tools turn the attendee row into a recognition tool, not just attendance: circular member photo, lifetime class-count milestone tags ('50th Class'), first-timer / VIP chips, new-client badges, and even personality-type colored avatar rings with coaching tips (Walla). This lets staff personalize the greeting at the moment of check-in.

Used by: Mariana Tek Walla GymMaster Mindbody

→ For Wix: Beyond a name, a Wix roster row can cheaply carry an avatar/initials, a first-timer or returning badge, and visit count from existing Bookings/CRM data — turning check-in into a light hospitality moment. Keep it to 1-2 chips on a phone row to avoid the density problem; reserve richer recognition for the tablet kiosk and desktop where there's room.

11

Time-window eligibility logic for door + booking-anchored check-in

Check-in/access is gated to a window around the booked session: barcode check-in only valid with a class booked within ~30 min (Glofox), Kisi door access within 59 min (Glofox) or a configurable before/after window (TeamUp), the member-app 'Check In' button only 'illuminated' inside the window (PushPress, Mariana Tek), and geofenced self check-in within a radius (Mariana Tek 100m default, Momence with a red out-of-range warning).

Used by: ABC Glofox TeamUp PushPress Mariana Tek Momence Mindbody

→ For Wix: Anchor check-in to the Bookings session time: enable the action a configurable window before start, auto-disable after, and for member self check-in consider a geofence/booking-window gate. Expose the window as a simple before/after-start setting (TeamUp's model is the clearest) and show owners a plain-language summary of the rule rather than raw config.

UI/UX takeaways for the Wix Check-in Gate

Concrete, mobile-first design implications for the Owner App check-in surface (Phase 1) and tablet kiosk (Phase 2).

  1. Make the existing Bookings booking list the check-in surface — do not build a separate screen. Put one thumb-reachable tap target on every attendee row that flips to a green checkmark; this is the single most universal pattern (all 11 competitors) and the natural mobile-first move for the Owner App.
  2. Standardize a 4-state status vocabulary used identically across roster, kiosk and door events: green = checked-in/granted, red = blocked/denied, amber = owing-but-allowed warning, grey = ineligible/expired. Always pair color with a text label (Kisi/GymMaster/Glofox) so it reads on a small screen and passes accessibility.
  3. Surface eligibility ON the row before the tap, not after: a chip for valid plan / remaining passes / expired / payment due / unsigned form. Follow TeamUp (greyed-but-visible ineligible option) and GymMaster (explicit human-readable denial reason + amount owed) rather than a generic post-tap failure.
  4. Treat the blocked case as a first-class inline flow: from the row, let the owner collect payment ('Check-in & Pay' — WellnessLiving), capture a signature in place (Walla/Mariana Tek), or override-and-check-in (TeamUp/PushPress) without leaving the roster. Tag override as an Owner-only power that a future kiosk mode disables.
  5. Add-a-walk-in should lead with one universal type-ahead field (name/phone/email, no mode toggle, results after ~2 characters — GymMaster) plus a '+' to create a new client mid-session (Mindbody/PushPress). Avoid forcing the owner to pick a search mode first.
  6. Anchor check-in to the session time window: enable the action a configurable interval before start and auto-disable it after (PushPress 'illuminated' button, TeamUp before/after-start model, Glofox 30/59-min windows). Show owners a plain-language rule summary, not raw config.
  7. Treat the Kisi door integration as a check-in source: a door unlock should post an arrival to the same booking and flip the same green check, with membership/booking status syncing in real time to auto-grant/revoke (Mindbody, Glofox, TeamUp). Owners should never reconcile door events by hand. Provide one 'allow owing/blocked members' toggle for the failed-payment edge case.
  8. Add a compact roster header: a fill/attendance counter (PushPress 3-up band) plus a status filter (All / Checked-in / Not arrived / No-show) and a bulk 'Check in all' for the whole-class-arrives case (Glofox 'Check-in Everyone', Vagaro 'Check-In All'). On phone use a horizontal chip row, not desktop tabs.
  9. For Phase-2 kiosk, reuse the Owner App roster in a PIN-locked Kiosk Mode (overrides off, restricted nav — PushPress/Mariana Tek) instead of a separate app; the kiosk confirmation should echo class/time/spot/remaining-passes before a single 'Done' (Mindbody resource, Mariana Tek spot+add-ons).
  10. Plan a QR/Apple Wallet member credential auto-generated from existing Wix contact data (Glofox auto-barcode from phone number, Vagaro QR + Wallet, GymMaster dynamic anti-fraud QR) so self check-in and the Kisi tap need no physical cards. Use a refreshing/dynamic QR to prevent sharing.
  11. Keep owner-side confirmation lightweight (inline toast + row turning green); reserve loud full-screen/audio confirmation (GymMaster green wash, WellnessLiving chime + tappable red reason) for the unattended kiosk and member self check-in where the actor is at arm's length.
  12. Enrich each row cheaply from existing Bookings/CRM data with avatar/initials, a first-timer or returning badge, and visit count to make check-in a light hospitality moment (Mariana Tek milestones, Walla badges) — but cap at 1-2 chips per row on a phone to control density; save richer recognition for tablet/desktop.

👤 Staff check-in flow

STAFF CHECK-IN (Mindbody Business app, Class & Course Sign-In screen): 1) Staff logs into the Mindbody Business app and taps "Schedule" in the bottom navigation. 2) Taps a class/course to open the Class & Course Sign-In screen (the roster). This screen lists everyone signed up, shows who has already signed in, and lets staff add a client, request a substitute teacher, cancel the class, or manage the waitlist. 3) To check a client in, staff taps the "Sign in" button to the LEFT of the client's name; a GREEN CHECKMARK then displays next to that client to confirm they are signed in (verbatim from Mindbody help docs: "tap the Sign in button to the left of the client's name, and a green checkmark displays"). 4) To add a walk-in / new attendee, staff taps the "+" (Plus) at top right; the Clients screen opens to search by name or Client ID, or "Add new client." 5) The roster surfaces eligibility/account status (remaining class passes / package balance, payment or membership issues) so staff can resolve or override before completing check-in. Staff permissions gate the screen: "Launch Sign-In screen" (Clients), plus "Make reservations" and "View reservations" (Classes & Courses). The same check-in is also available from the desktop browser front-desk "Client Check In" sign-in screen. With door-access integrations, door-open events report back into Mindbody and can auto-mark class/appointment attendance, so a member entering the building auto-checks-in.

🙋 Client / self check-in & door flow

CLIENT SELF CHECK-IN (Mindbody Check-In app on iPad/tablet kiosk): 1) The business signs into the dedicated Mindbody Check-In app on a tablet at the front desk ("All you need is a tablet... download the app, sign in, and you're ready") and runs it in self-check-in (kiosk) mode; settings are password-protected. 2) The client walks up and self-identifies. Options: (a) type their NAME (search results appear after only a few characters); (b) enter their Client ID on a station with a keyboard; (c) scan a barcode from a key tag / plastic card / branded-app digital membership card using a connected barcode scanner (Socket Mobile scanners supported). 3) The client taps to sign into class / check in with ONE TAP ("one tap customer check-in"). 4) On check-in, the screen displays the RESOURCE for the visit (e.g., room or stylist's chair) so the client knows where to go without staff guiding them, shows the number of REMAINING VISITS/passes in their package, alerts them when it is time to buy more, and the software deducts the visit from the relevant pricing option. 5) Staff get a notification when a client arrives, or when an account issue (insufficient funds / no remaining classes / needs assistance) needs front-desk help. The browser-based SELF SIGN IN / Self Check In screen supports the same barcode-scan / Client-ID flow and is used to sign into classes, mark arrival for appointments/courses, or check in on a facility visit. CONSUMER SELF CHECK-IN: clients can also sign themselves into a pre-booked class from the Mindbody consumer app (or branded app) on their own phone. DOOR ACCESS (Kisi / Gym Access / Brivo): members unlock the door by tapping their phone on a reader, an Apple/Google Wallet pass, a Brivo Mobile Pass, or a fob/card/Mindbody barcode. Access is granted in real time to paying members and instantly activated/revoked on membership change (missed payment / expiry); it can be scoped to opening-hour schedules or booked time slots, supports unstaffed-hours and multi-location access, syncs usage back to Mindbody, and can auto-record arrivals.

UI/UX highlights

  • Tap-to-check-in: staff tap a 'Sign in' button to the LEFT of each client's name on the roster; a green checkmark appears as the signed-in confirmation state (per Mindbody help docs)
  • Self-check-in kiosk centers on a single 'one tap customer check-in' to avoid the 'front desk traffic jam' (vendor app-store/marketing copy)
  • Type-ahead name search: results surface after only a few characters instead of requiring the full name plus a search tap
  • On check-in the screen tells the client which RESOURCE (room / stylist's chair) to go to, removing the need for staff to direct them
  • Remaining visit/package balance is shown to the client at check-in with a low-balance alert prompting them to buy more; the visit is auto-deducted from their pricing option
  • Staff arrival notifications fire on check-in and on account issues (insufficient funds / no remaining classes / needs assistance)
  • Password-protected kiosk settings and customizable front-desk alerts lock down the unattended station; runs on just a tablet from the App Store / Google Play
  • Multiple parallel check-in surfaces (front-desk Business app, dedicated Check-In kiosk, browser Self Sign In, consumer/branded app, door reader) all feed the same roster
  • Door-access integrations auto-convert a door-open event into an 'arrival' and can auto-mark class/appointment attendance, removing a manual check-in step
  • Membership-driven access: entry instantly granted/revoked on membership change; access scopable to opening hours or booked slots; supports unstaffed-hours and multi-location entry
  • '+' (Plus) at top right adds a walk-in / new client into the roster mid-class (search by name or Client ID, or 'Add new client')

Visual style

The Mindbody Check-In kiosk uses a clean, low-density, single-purpose layout tuned for a tablet at arm's length: a large central search/name field, big tappable list rows, and a prominent one-tap action so a client (or busy front-desk staff) can act without precision. The brand palette skews to Mindbody's purple/indigo accents on a light/white background. The Business-app roster is a vertical list of clients with a per-row left-aligned 'Sign in' control; status is communicated primarily through a green checkmark (signed-in) versus an unchecked row, plus inline package/eligibility info and (after check-in) the assigned resource. Status treatment is binary and color-coded (green = checked in; distinct alert states for account/payment issues). Marketing imagery presents the app in phone and tablet device frames with short benefit caption cards ('Check in clients with a click', 'Sync your staff schedule', plus 'effortless' and 'control' themed cards on the Check-in app page). The door-access (Kisi/Gym Access/Brivo) surface is hardware-plus-mobile: a phone tapping a wall-mounted reader, dashboard tiles for access events, and badge/credential visuals rather than dense data tables.
